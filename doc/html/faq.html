Frequently Asked Questions

1. Why this project name ?

CCruncher calculates the credit portfolio risk valuation with intensive numerical computation. These kind of programs are called number crunchers. The mix between the terms 'credit' and 'number cruncher' generates the name 'credit cruncher'. CCruncher is not related to the recent credit crunch, but is a tool that can aid to control the risks involved in this crisis.

3. What is the project's maturity ?

ccruncher-2.2 is a stable version. If you have any bug or suggestion to report, please send an email to ccruncher team or post it to SourceForge.

4. Is there a company that provides support for CCruncher ?

Yes. Tatine.es provides consulting services for the analysis of SME loan portfolios using CCruncher. It also provides support services such as integration, maintenance and ad-hoc development related to CCruncher.

5. What about performance ?

CCruncher is designed to be extremely fast. When they are 10 obligors it computes 1000000 simulations in few seconds. When obligors are 50000 it takes few minutes to do the same task. CCruncher execution time depends on multiple factors: machine (processor, L1/L2 caches, cores, etc.), number of simulations, number of obligors, number of ratings, number of factors, number of segmentations, and number of assets.

6. Why can the input files be gzipped (*.gz) but not zipped (*.zip) ?

The zip algorithm is covered by patents, but gzip is not; furthermore gzip is open source. You can download a gzip tool for windows from 7-zip. UNIX environments have the gzip/gunzip commands out of the box. You can read more at the gzip website.

7. Can an obligor belong to a weighted sum of factors ?

Currently not. CCruncher was designed to work with portfolios of SMEs. Usually, in these portfolios each obligor belongs to a single sector/factor. We can add this feature if there are several people who request it.

8. I don't have the transition matrix. What can I do ?

You can assume that the default probability over time for each rating is an exponential distribution. In this case set all the transition matrix values to 0 except the diagonal values, that contains the probability of non-default, (1-pr), and the 'default' column values that contains the probability of default, pr.

9. Some samples exhibits strange behavior. What is happening ?

1) Standard Error of VaR increase when the number of simulations increases: We use the Maritz-Jarret to obtain these values. To reduce the computational cost we truncate the involved sum. In some cases, this action causes the observed behavior.
2) VaR plot has jumps: This happens because the resulting portfolio loss distribution is discrete with a support composed of few values.
