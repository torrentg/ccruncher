Overview

CreditCruncher valora el riesgo de impago de una cartera de créditos usando la técnica de simulación Monte Carlo.

Keywords: CreditMetrics, Sectorial Correlation, Copula, Monte Carlo, VAR (Value At Risk), Ratings, Transition Matrix

La metodología usada es muy parecida a la descrita por CreditMetrics. Se dispone de una cartera de N clientes donde cada cliente tiene contratado uno o varios productos con riesgo de crédito. Cada cliente tiene asignado un rating de calidad crediticia y existe una matriz de transición que permite determinar la probabilidad de fallido a un horizonte de tiempo fijado. Los clientes pertenecen a diversos sectores de los que disponemos de una matriz de correlación que indica el grado de dependencia intersectorial en caso de fallido. A partir de la matriz de correlación intersectorial se construye la matriz de correlación entre clientes. Finalmente se genera un conjunto de N variables aleatorias uniformes correlacionadas según esta matriz (cópula). Se usa la matriz de transición para determinar la evolución del rating inicial de cada cliente y se evalua el valor de sus productos. Si se repite este proceso un número elevado de veces disponemos de un conjunto de valores posibles de la cartera que permiten determinar la distribución del valor de la cartera y calcular el VAR (Value At Risk). Para mas información consúltese el manual de CreditCruncher donde se describe con detalle todos los pasos realizados.

En la implementación de CreditCruncher se consideran como claves los factores que se detallan y justifican a continuación:

Rendimiento. El número de clientes y/o activos de la cartera puede ser importante (puede ser del orden de 50.000 activos). En este caso el número de operaciones a realizar es masivo así como la memoria que puede llegar a necesitarse. Por este motivo el lenguage de programación escogido es C++. Además se proporciona soporte para la computación distribuida a través de LAM/MPI.

Fiabilidad. Se pretende valorar el riesgo de crédito sin caer en otro tipo de riesgos (pe. riesgo de modelo o riesgo operacional). Por este motivo el código es abierto y la metodología descrita está documentada. A nivel de codificación se recurre a proyectos vigentes de prestigio para resolver problemas especificos (manipulación de matrices, interpretación de ficheros xml, generación de variables aleatorias, etc.). Además se hace hincapié en una correcta codificación basada en la creación de tests unitarios gestionados por la suite cppunit y durante el desarrollo se garantiza el comportamiento esperado a través de la herramienta valgrind. Para mas información consultar la sección related projects.

Comodidad. La entrada de datos se realiza a través de un xml del que se dispone de una dtd y se encuentra documentado (véase la sección tag reference). Esto debe permitir a un usuario no técnico definir el problema a resolver de una forma cómoda y estándar. El resultado consiste en un fichero plano que puede ser tratado con facilidad en una hoja de cálculo.

CreditCruncher está desarrollado y probado en Linux. El motivo de esta decisión es la fiabilidad, licencia y soporte para el procesamiento paralelo (lam/mpi). El port a otros sistemas tipo unix no tiene que presentar problemas. Si lo realiza no dude en informar.
