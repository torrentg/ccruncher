CreditCruncher XML input tag reference

This document describe the xml input file for CreditCruncher-0.2. In the distribution package you can find the DTD and some XML samples into the directory $CCRUNCHER/samples.

params

This sections contains the basic properties of the problem to be solved. Properties can be unordered.

Example

            
  <params>
    <property name="time.begindate" value="18/02/2003"/>
    <property name="time.steps" value="12"/>
    <property name="time.steplength" value="2"/>
    <property name="stopcriteria.maxiterations" value="3000"/>
    <property name="stopcriteria.maxseconds" value="30000000"/>
    <property name="copula.type" value="normal"/>
    <property name="copula.seed" value="387650932874"/>
    <property name="montecarlo.antithetic" value="false"/>
    <property name="portfolio.onlyActiveClients" value="true"/>
  </params>

property Attributes

Name Description Type
time.begindate first day of the simulation. Initial ratings are refered to this date. The format of this field is: DD/MM/YYYY. date
time.steps number of time steps. Fix this number with caution since has a big impact in the number of calculations to be performed. int
time.steplength length of each time step in natural months. We suggest to put low values (pe. 1, 3 o 6). int
stopcriteria.maxiterations maximum number of iterations in Monte Carlo algorithm.
ccruncher stops when one of stop criteria is achieved. 		long
stopcriteria.maxseconds maximum number of seconds in current execution.
ccruncher stops when one of stop criteria is achieved. 		long
copula.type type of copula used to generate correlated variables. At this moment exist only one implementation, the normal copula.
accepted values: 'normal'. 		string
copula.seed seed used for initialize random number generator. If this property is not set, the used seed will depend of cpu clock. long
montecarlo.antithetic tell to use variance reduction technique named antithetic in Monte Carlo. If you doubt, type 'false', otherwise 'true'. boolean
portfolio.onlyActiveClients if you desire simulate all clients, even though clients without risk in analized period, type 'false'. If you desire simulate only active clients in the simulation period type 'true'. We suggest to write always 'true' due to the reduction in the number of calculations. boolean

Interests

This section contain the interest curves used to transport money-values along the time. The definition of curve 'spot' is required.

Example

  
  <interests>
    <interest name="spot" date="18/11/2004">
      <rate t="0" r="0.0"/>
      <rate t="1" r="0.04"/>
      <rate t="2" r="0.041"/>
      <rate t="3" r="0.045"/>
      <rate t="6" r="0.0455"/>
      <rate t="12" r="0.048"/>
      <rate t="24" r="0.049"/>
      <rate t="60" r="0.05"/>
      <rate t="120" r="0.052"/>
    </interest>
  </interests>

Interest attributes

Name Description Type
name name of the interest type string
date date of the interest curve. Generally match the begin simulation date (but is not required). The format of this field is: DD/MM/YYYY. date

Rate attributes

Name Description Type
t time (in days) from date of the interest curve. If is needed a value between two given dates, the value will be interpolated (linear). int
r interest rate at time t. Is a value between -0.5 and 1.0. (per example, a rate of 2.5% will be writed as 0.025). double

Ratings

This section contains the defined ratings.

Example

            
  <ratings>
    <rating name="A" order="1" desc="excelent"/>
    <rating name="B" order="2" desc="good"/>
    <rating name="C" order="3" desc="medium"/>
    <rating name="D" order="4" desc="bad"/>
    <rating name="E" order="5" desc="default"/>
  </ratings>

Rating Attributes

Name Description Type
name rating indentification. All references to rating (pe. client rating) are made using this field. string
order priority of this rating. the best rating have value 1, next 2, etc. (sequencially). The default rating always have the highest value. int
desc rating description. This value only is used in reports. this description would be unique. string

Transition Matrix

This section contains the used transition matrix. ccrucnher checks that sum of elements of each row is 1.0. Too is checked that only exist a row with all probability concentred in a unique element. The number of elements to define is NxN, where N is the number of ratings.

Example


  <mtransitions period="1" epsilon="1e-12">
    <transition from="A" to="A" value="0.80"/>
    <transition from="A" to="B" value="0.10"/>
    <transition from="A" to="C" value="0.07"/>
    <transition from="A" to="D" value="0.02"/>
    <transition from="A" to="E" value="0.01"/>
    <transition from="B" to="A" value="0.05"/>
    <transition from="B" to="B" value="0.75"/>
    <transition from="B" to="C" value="0.10"/>
    <transition from="B" to="D" value="0.07"/>
    <transition from="B" to="E" value="0.03"/>
    <transition from="C" to="A" value="0.03"/>
    <transition from="C" to="B" value="0.07"/>
    <transition from="C" to="C" value="0.70"/>
    <transition from="C" to="D" value="0.14"/>
    <transition from="C" to="E" value="0.06"/>
    <transition from="D" to="A" value="0.05"/>
    <transition from="D" to="B" value="0.05"/>
    <transition from="D" to="C" value="0.15"/>
    <transition from="D" to="D" value="0.60"/>
    <transition from="D" to="E" value="0.15"/>
    <transition from="E" to="A" value="0.00"/>
    <transition from="E" to="B" value="0.00"/>
    <transition from="E" to="C" value="0.00"/>
    <transition from="E" to="D" value="0.00"/>
    <transition from="E" to="E" value="1.00"/>
  </mtransitions>

MTransitions Attributes

Name Description Type
period number of years that transition matrix cover. double
epsilon value used to determine if a value is 0.0 or not in performed checks. must be a value near to 0. We sugest a value very near to 0.0, pe. 1E-12. double

Transition Attributes

Name Description Type
from initial rating. would be a rating identifier (see name attribute in rating tag) string
to final rating. would be a rating identifier (see name attribute in rating tag). string
value probability of rating migration (a value between 0 and 1) double

Sectors

This section contains the defined sectors.

Example

            
  <sectors>
    <sector name="S1" order="1" desc="oil and gas"/>
    <sector name="S2" order="2" desc="other sectors"/>
  </sectors>

Sector Attributes

Name Description Type
name sector identification. All references to sector (pe. client sector) are made using this field. string
order priority of this sector. This number is a sequence that begins at 1. int
desc sector description. This value only is used in reports. this description would be unique. string

Correlations

This section contains the correlation matrix between sectors. Correlation matrix is a simetric and definite positive matrix. You only need to enter the upper triangular elements. ccruncher checks that every element is between -1.0 and 1.0 and that is definite positive. The number of elements to define is (nsectors+1) x nsectors / 2.

Example

  
  <mcorrels epsilon="1e-12">
    <sigma sector1="S1" sector2="S1" value="0.2"/>
    <sigma sector1="S1" sector2="S2" value="0.1"/>
    <sigma sector1="S2" sector2="S2" value="0.25"/>
  </mcorrels>

MCorrels Attributes

Name Description Type
epsilon value used to determine if a value is 0.0 or not in performed checks. must be a value near to 0. We sugest a value very near to 0.0, pe. 1E-12. double

Sigma Attributes

Name Description Type
sector1 would be a sector identifier (see name attribute in sector tag) string
sector2 would be a sector identifier (see name attribute in sector tag) string
value correlation of defaults between sector1 and sector2.
Correlation(defaults(S1), defaults(S2)) 		double

Segmentations

This sections contains the defined segmentations. Segmentations allow analyze risk portfolio according to jerarquical organization or other type of segmentation. A segmentation is composed by segments. Exist 2 types of segmentations in function the class of elements contained; client segmentations (pe. sector, client size, initial rating, etc.) and asset segmentations (pe. products, branch where the contract is made, etc.). All segmentations will be completes (one element only belongs to one segment and the union of all segments give the total portfolio). Every segmentation have defined (in impliced form) a segment named 'rest'. If a client (or a asset) belongs nowhere, then belongs to 'rest' segment. Finally, exist 2 special segmentations, 'client' (one segment per client) and 'asset' (one segment per asset) where you don't need to specify all segment, these are defined in implicit form.

Example

  
  <segmentations>
    <segmentation name="portfolio" components="asset"/>
    <segmentation name="client" components="client">
      <segment name="*"/>
    </segmentation>
    <segmentation name="asset" components="asset">
      <segment name="*"/>
    </segmentation>
    <segmentation name="sector" components="client">
      <segment name="S1"/>
      <segment name="S2"/>
    </segmentation>
    <segmentation name="size" components="client">
      <segment name="big"/>
      <segment name="medium"/>
    </segmentation>
    <segmentation name="product" components="asset">
      <segment name="bond"/>
    </segmentation>
    <segmentation name="office" components="asset">
      <segment name="0001"/>
      <segment name="0002"/>
      <segment name="0003"/>
      <segment name="0004"/>
    </segmentation>
  </segmentations>

Segmentation Attributes

Name Description Type
name segmentation identification. All references to segmentation (pe. belongs-to tag) are made using this field. string
components class of components for the segments. Accepted values are 'client' and 'asset' string

Segment Attributes

Name Description Type
name segment name. Different segmentations can have segments with the same name. All references to segment (pe. belongs-to tag) are made using this field (always in conjuntion with segmentation). If a segmentation haven't segments, all elements belongs to segment 'rest'. 'client' and 'asset' segmentations allow the segment '*' to indicate that they have many segments as clients (or assets) exist. string

Aggregators

This section contains defined aggregations. An aggregation is a simulation values recopilation over a segmentation. Be cautious about the information level to compile (have a important impact in memory and disk usage). Each aggregator-segment have its own output file, if you have 50.000 clients and you define the segmentation 'client' or 'asset' and an aggregator over these, you obtain more than 50.000 files in the disk.

Example


  <aggregators>
    <aggregator name="aggregator1" segmentation="portfolio" 
                type="values" full="true"/>
    <aggregator name="aggregator2" segmentation="client" 
                type="values" full="false"/>
    <aggregator name="aggregator3" segmentation="client" 
                type="ratings" full="false"/>
    <aggregator name="aggregator4" segmentation="sector" 
                type="values" full="false"/>
    <aggregator name="aggregator5" segmentation="sector" 
                type="ratings" full="false"/>
  </aggregators>

aggregator1 list portfolio values (one per simulation). In this case, full attribute says that we want all the values, not only the average and stddev. aggregator2 and aggregator3 gives summarized information for each client (average and stddev for values and ratings). aggregator4 and aggregator5 gives the same info for sectors.

Aggregator Attributes

Name Description Type
name unique aggregator name string
segmentation segmentation name connected to this aggregator string
type type of aggregation performed. Accepted values are:
values. if segmentation components are assets, results are values from assets belonging to segmentation , if segmentation components are clients, results are values from clients (sum of client assets) belonging to segmentation.
ratings. results are ratings from clients. This type only can be used when segmentation components is 'client'.
 string
full detail level. If true, all values will be listed. Otherwise, average and stddev is listed for values and number of ocuerrences per defined rating for ratings. Use this porperty with caution (have a important impact in memory and disk usage). boolean

Portfolio

This section contains the defined portfolio.

Example

  
  <portfolio>
    <client rating="A" sector="S1" name="Repsol" id="000001">
      <belongs-to concept="sector" segment="S1"/>
      <belongs-to concept="size" segment="medium"/>
      <asset class="bond" id="op1">
        <belongs-to concept="product" segment="bond"/>
        <belongs-to concept="office" segment="0001"/>
        <data>
          <values at="01/06/2002" cashflow="10.0" exposure="1500.0" recovery="2000.0" />
          <values at="01/01/2003" cashflow="10.0" exposure="1500.0" recovery="2000.0" />
          <values at="01/06/2003" cashflow="10.0" exposure="1500.0" recovery="2000.0" />
          <values at="01/01/2004" cashflow="10.0" exposure="1500.0" recovery="2000.0" />
          <values at="01/06/2004" cashflow="10.0" exposure="1500.0" recovery="2000.0" />
        </data>
      </asset>
      <asset class="bond" id="op2">>
        <belongs-to concept="office" segment="0004"/>
        <belongs-to concept="product" segment="bond"/>
        <data>
          <values at="01/01/2000" cashflow="10.0" exposure="1500.0" recovery="2000.0" />
          <values at="01/06/2000" cashflow="10.0" exposure="1500.0" recovery="2000.0" />
          <values at="01/01/2001" cashflow="10.0" exposure="1500.0" recovery="2000.0" />
          <values at="01/06/2001" cashflow="10.0" exposure="1500.0" recovery="2000.0" />
          <values at="01/01/2002" cashflow="10.0" exposure="1500.0" recovery="2000.0" />
        </data>
      </asset>
    </client>
    <client rating="B" sector="S2" name="ATT" id="000002">
      <belongs-to concept="sector" segment="S2"/>
      <belongs-to concept="size" segment="big"/>
      <asset class="bond" id="op3">
        <belongs-to concept="office" segment="0003"/>
        <belongs-to concept="product" segment="bond"/>
        <data>
          <values at="01/01/2000" cashflow="10.0" exposure="1500.0" recovery="2000.0" />
          <values at="01/06/2000" cashflow="10.0" exposure="1500.0" recovery="2000.0" />
          <values at="01/01/2001" cashflow="10.0" exposure="1500.0" recovery="2000.0" />
          <values at="01/06/2001" cashflow="10.0" exposure="1500.0" recovery="2000.0" />
          <values at="01/01/2002" cashflow="10.0" exposure="1500.0" recovery="2000.0" />
        </data>
      </asset>
    </client>
  </portfolio>

Observe how client haven't defined a 'belong-to' to 'client' segmentation. This is performed implicitly (the same occurs with 'asset'). If you try to asign a client to a segmentation of asset components (or viceversa), the input file will be rejected.

Client Attributes

Name Description Type
rating initial client rating at begin simulation date string
sector sector where this client is included string
name client name. only used for report string
id unique client identificator string

Belongs-To Attributes

Name Description Type
concept segmentation identifier (see segmentations) string
segment segment identifier (see segmentations) string

Asset Attributes

Name Description Type
name a descriptive name of the asset. Don't need to be unique string
id unique asset identifier string

Values Attributes

Name Description Type
at date (in format DD/MM/YYYY) where take place the values date
cashflow cashflow at date double
exposure exposure at date double
recovery recovery at date double
