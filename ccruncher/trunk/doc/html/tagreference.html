CreditCruncher XML tag reference

La versión en curso del documento xml descrito en esta sección es la 0.1. La distribución de CreditCruncher contiene la versión de la DTD y un fichero XML de ejemplo. La DTD es el conjunto de instrucciones que indican como se espera que esté creado el fichero XML.

params

Este elemento contiene las propiedades básicas del problema a resolver. Deben especificarse todas y cada una de ellas. No es necesario mantener el orden de las propiedades.

Example

            
  <params>
    <property name="time.begindate" value="18/02/2003"/>
    <property name="time.steps" value="12"/>
    <property name="time.steplength" value="2"/>
    <property name="stopcriteria.maxiterations" value="3000"/>
    <property name="stopcriteria.maxseconds" value="30000000"/>
    <property name="copula.type" value="normal"/>
    <property name="copula.seed" value="387650932874"/>
    <property name="montecarlo.antithetic" value="false"/>
    <property name="portfolio.onlyActiveClients" value="true"/>
  </params>

property Attributes

Name Description Type
time.begindate indica el primer día de la simulación. Los ratings iniciales de los clientes son a este día. El formato de este campo debe ser de la forma DD/MM/YYYY. date
time.steps número de pasos en el tiempo. Fije este parámetro con cuidado ya que tiene un impacto importante en el número de operaciones a realizar. int
time.steplength longitud de cada paso temporal en meses naturales. Se recomienda poner valores bajos (pe. 1, 3 o 6) dependiendo del periodo de simulación abarcado (periodos pequeños, pasos pequeños). int
stopcriteria.maxiterations número máximo de iteraciones del método de Monte Carlo a realizar. Se dará el problema por resuelto cuando sea satisfecho alguno de los criterios de parada. long
stopcriteria.maxseconds número máximo de segundos en tiempo absoluto (no tiempo de cpu). Se dará el problema por resuelto cuando sea satisfecho alguno de los criterios de parada. long
copula.type tipo de cópula utilizado para generar las variables aleatorias correlacionadas. En este momento solamente existe una implementación, la cópula archimediana o normal ('normal'). string
copula.seed semilla utilizada para la generación de números aleatorios utilizados en la cópula. En caso de no especificarse esta propiedad se inicializará con un valor que depende de la hora en el momento de ejecutarse. long
montecarlo.antithetic indica si debe utilizarse la técnica de reducción de varianza antithetic en el método de Monte Carlo. Si tiene dudas sobre el significado o utilidad de esta técnica, indique false. boolean
portfolio.onlyActiveClients indica si deben simularse todos los clientes (a pesar de no tener riesgo contratado) o pro el contrario, simular solamente aquellos clientes que tienen riesgo contratado. Recomendamos poner siempre true ya que supone un ahorro importante de tiempo de cómputo en caso de tener clientes inactivos. boolean

Interests

Este apartado contiene los tipos de interés usados por los productos financieros (en caso de depender de estos). Es obligatorio la definición del tipo de interés 'spot' que se usará para transportar los flujos de efectivo a lo largo del tiempo.

Example

  
  <interests>
    <interest name="spot" date="18/11/2004">
      <rate t="0" r="0.0"/>
      <rate t="1" r="0.04"/>
      <rate t="2" r="0.041"/>
      <rate t="3" r="0.045"/>
      <rate t="6" r="0.0455"/>
      <rate t="12" r="0.048"/>
      <rate t="24" r="0.049"/>
      <rate t="60" r="0.05"/>
      <rate t="120" r="0.052"/>
    </interest>
  </interests>

Interest attributes

Name Description Type
name nombre del tipo de interés. Todas las referencias al tipo de interés definido deberán realizarse indicando este nombre. Siempre debe existir el tipo de interés 'spot'. string
date fecha en la que se ha determinado el tipo de interés. Generalmente coincide con la fecha de inicio de la simulación. El formato de este campo debe ser DD/MM/YYYY. date

Rate attributes

Name Description Type
t tiempo en días respecto de la fecha inicial del tipo de interés. En caso de ser necesario un valor de tiempo no indicado se realizará una interpolación lineal usando los valores disponibles. int
r tipo de interés en tanto por uno en la fecha indicada por t. Debe ser un valor entre -0.5 y 1.0. double

Ratings

Este apartado contiene la definición de los ratings usados.

Example

            
  <ratings>
    <rating name="A" order="1" desc="excelent"/>
    <rating name="B" order="2" desc="good"/>
    <rating name="C" order="3" desc="medium"/>
    <rating name="D" order="4" desc="bad"/>
    <rating name="E" order="5" desc="default"/>
  </ratings>

Rating Attributes

Name Description Type
name identificador del rating. Todas las referencias al rating definido deberán realizarse mediante este identificador. string
order orden de prioridad del rating. El mejor debe tener el valor 1, el siguiente el valor 2, etc. El rating 'fallido' siempre debe ser el último. Debe tratarse de una ordenación completa y siempre empezar por el valor 1. int
desc descripción del rating. Este valor solamente se usa para finalidades de presentación de la información. Solamente tiene valor descriptivo. A pesar de esto, debe rellenarse y no puede repetirse la descripción. string

Transition Matrix

Esta sección contiene la definición de la matriz de transición utilizada. Se comprobará que se cumple que al suma de los elementos de cada fila suma 1.0. También se comprueba que existe una única fila en la que toda la probabilidad se concentrada en un elemento (corresponde al fallido). El número de elementos a definir es el número de ratings al cuadrado.

Example


  <mtransitions period="1" epsilon="1e-12">
    <transition from="A" to="A" value="0.80"/>
    <transition from="A" to="B" value="0.10"/>
    <transition from="A" to="C" value="0.07"/>
    <transition from="A" to="D" value="0.02"/>
    <transition from="A" to="E" value="0.01"/>
    <transition from="B" to="A" value="0.05"/>
    <transition from="B" to="B" value="0.75"/>
    <transition from="B" to="C" value="0.10"/>
    <transition from="B" to="D" value="0.07"/>
    <transition from="B" to="E" value="0.03"/>
    <transition from="C" to="A" value="0.03"/>
    <transition from="C" to="B" value="0.07"/>
    <transition from="C" to="C" value="0.70"/>
    <transition from="C" to="D" value="0.14"/>
    <transition from="C" to="E" value="0.06"/>
    <transition from="D" to="A" value="0.05"/>
    <transition from="D" to="B" value="0.05"/>
    <transition from="D" to="C" value="0.15"/>
    <transition from="D" to="D" value="0.60"/>
    <transition from="D" to="E" value="0.15"/>
    <transition from="E" to="A" value="0.00"/>
    <transition from="E" to="B" value="0.00"/>
    <transition from="E" to="C" value="0.00"/>
    <transition from="E" to="D" value="0.00"/>
    <transition from="E" to="E" value="1.00"/>
  </mtransitions>

MTransitions Attributes

Name Description Type
period número de años que abarca la matriz de transición. double
epsilon valor usado para determinar si un valor se puede considerar 0.0 o no. Debe ser un valor positivo cercano a cero. Se recomienda poner un valor muy cercano a 0.0 (pe. 1E-12). double

Transition Attributes

Name Description Type
from rating de partida. Debe ser un identificador de rating definido en el apartado de ratings. string
to rating destino. Debe ser un identificador de rating definido en el apartado de ratings. string
value probabilidad del evento (en tanto por uno). double

Sectors

Este apartado contiene la definición de los sectores usados.

Example

            
  <sectors>
    <sector name="S1" order="1" desc="oil and gas"/>
    <sector name="S2" order="2" desc="other sectors"/>
  </sectors>

Sector Attributes

Name Description Type
name identificador del sector. Todas las referencias al sector definido deberán realizarse mediante este identificador. string
order orden de prioridad del sector. Debe tratarse de una ordenación completa y siempre empezar por el valor 1. int
desc descripción del sector. Este valor se usa para finalidades de presentación de la información. Solamente tiene valor descriptivo. A pesar de esto, debe rellenarse y no puede repetirse la descripción. string

Correlations

Esta sección contiene la definición de la matriz de correlación entre los sectores definidos. Se trata de una matriz simétrica definida positiva. Solamente debe indicarse la diagonal superior o inferior. Se comprobará que cada elemento de la matriz se encuentra entre -1.0 y 1.0 y que esta es definida positiva. El número de elementos a definir es: (nsectors+1) x nsectors / 2.

Example

  
  <mcorrels epsilon="1e-12">
    <sigma sector1="S1" sector2="S1" value="0.2"/>
    <sigma sector1="S1" sector2="S2" value="0.1"/>
    <sigma sector1="S2" sector2="S2" value="0.25"/>
  </mcorrels>

MCorrels Attributes

Name Description Type
epsilon valor usado para determinar si un valor se puede considerar 1.0 o no. Debe ser un valor positivo cercano a cero. Se recomienda poner un valor muy cercano a 0.0 (pe. 1E-12). double

Sigma Attributes

Name Description Type
sector1 sector implicado en la correlación. Debe tratarse de un sector definido en el apartado de sectores. string
sector2 sector implicado en la correlación. Debe tratarse de un sector definido en el apartado de sectores. string
value correlación del fallido entre sector1 y sector2 en tanto por 1. double

Segmentations

Esta sección contiene las segmentaciones definidas. La finalidad de las segmentaciones es permitir analizar el riesgo de la cartera según el nivel de detalle u organización jerarquica del usuario sin que esta se encuentre prefijada. Existen dos tipos de segmentaciones (en función del tipo de elementos que contengan); segmentaciones de clientes (pe. por sector, tamaño del cliente, rating inicial, etc.) y segmentaciones de activos (pe. por tipo de activo, por oficina de contratación, etc.). Todas las segmentaciones deben ser completas (un elemento solamente puede pertenecer a un segmento y la unión de todos los elementos de los segmentos debe ser el total). Todas las segmentaciones tienen definido de forma implícita el elemento 'resto' que garantiza que si un elemento no tiene definida su pertenencia a la segmentación analizada, entonces pertenece al segmento 'resto'. Finalmente, existen 2 segmentaciones definidas por defecto que son 'client' (donde los segmentos definidos son los clientes existentes) y 'asset' (donde los segmentos definidos son los activos existentes). En estas segmentaciones no es necesario definir de forma explícita los segmentos definidos.

Example

  
  <segmentations>
    <segmentation name="portfolio" components="asset"/>
    <segmentation name="client" components="client">
      <segment name="*"/>
    </segmentation>
    <segmentation name="asset" components="asset">
      <segment name="*"/>
    </segmentation>
    <segmentation name="sector" components="client">
      <segment name="S1"/>
      <segment name="S2"/>
    </segmentation>
    <segmentation name="size" components="client">
      <segment name="big"/>
      <segment name="medium"/>
    </segmentation>
    <segmentation name="product" components="asset">
      <segment name="bond"/>
    </segmentation>
    <segmentation name="office" components="asset">
      <segment name="0001"/>
      <segment name="0002"/>
      <segment name="0003"/>
      <segment name="0004"/>
    </segmentation>
  </segmentations>

Segmentation Attributes

Name Description Type
name nombre de la segmentación. Todas las referencias a la segmentación definida deberán realizarse mediante este identificador. string
components tipo de componentes de los segmentos definidos. Solamente se aceptan los valores 'client' (para clientes) y 'asset' para activos. string

Segment Attributes

Name Description Type
name nombre de la segmento. Diferentes segmentaciones pueden tener segmentos definidos con el mismo nombre. Todas las referencias al segmento definido deberán realizarse mediante este identificador (siempre junto con el de segmentación). En caso no definir ningún segmento todos los elementos pertencen al segmento 'resto' (es el tipo de segmentación que utilizaremos si se desea obtener el valor de la cartera). Las segmentaciones 'client' y 'asset' admiten el segmento '*' para indicar que contienen tantos segmentos como elementos de tipo client o asset. string

Aggregators

Esta sección contiene las agregaciones que deben realizarse en el momento de realizar la simulación. Una agregación siempre se realiza sobre una segmentación definida y debe detallarse el nivel de información recopilada por cada segmento. Debe seleccionarse cuidadosamente el nivel de información a recopilar ya que puede tener un impacto importante en memoria utilizada y espacio de disco necesario.

Example


  <aggregators>
    <aggregator name="aggregator1" segmentation="portfolio" 
                type="values" full="true"/>
    <aggregator name="aggregator2" segmentation="client" 
                type="values" full="false"/>
    <aggregator name="aggregator3" segmentation="client" 
                type="ratings" full="false"/>
    <aggregator name="aggregator4" segmentation="sector" 
                type="values" full="false"/>
    <aggregator name="aggregator5" segmentation="sector" 
                type="ratings" full="false"/>
  </aggregators>

El aggregador aggregator1 se define para listar los valores simulados de la cartera; Se le indica el atributo full para poder determinar la distribución de los valores de la cartera. Aggregator2 y aggregator3 proprociona información resumida de cada cliente (valor medio y desviación estándar de los activos y ratings obtenidos). Aggregator4 y aggregator5 proprocionan la misma información a nivel de sector.

Aggregator Attributes

Name Description Type
name nombre de la agregación. Todas las referencias a la agregación definida se realizarán mediante este identificador. string
segmentation nombre de la segmentación sobre la que opera el agregador. Debe ser un identificador de segmentación definido en el apartado de segmentaciones. string
type tipo de agregación a realizar. Debe ser uno de los valores siguientes:
values. los valores agregados son el valor de los activos de los componentes de la segmentación. Si los componentes son de tipo cliente, se consideraran los activos de cada cliente.
ratings. los valores agregados son los ratings de los clientes de la segmentación. Este valor solamente puede ponerse cuando los componentes de la segmentación sean de tipo client.
 string
full indicador del detalle de los resultados a generar. En caso de seleccionar true se listarán todos los valores por cada segmento, en caso contrario se proprociona información resumida (media y desviación estándar en el caso de valores y número de casos en el caso de los ratings). Use esta propiedad con cuidado ya que puede generar una salida masiva de resultados y ralentizar la ejecución del proceso. boolean

Portfolio

Esta sección contiene la definición de la cartera que se pretende evaluar. Una cartera está compuesta de un conjunto de clientes cada uno de los cuales puede disponer de un conjunto de activos.

Example

  
  <portfolio>
    <client rating="A" sector="S1" name="Repsol" id="000001">
      <belongs-to concept="sector" segment="S1"/>
      <belongs-to concept="size" segment="medium"/>
      <asset class="bond" id="op1">
        <belongs-to concept="product" segment="bond"/>
        <belongs-to concept="office" segment="0001"/>
        <property name="issuedate" value="1/1/2000"/>
        <property name="term" value="120"/>
        <property name="nominal" value="1000.0"/>
        <property name="rate" value="0.06"/>
        <property name="ncoupons" value="10"/>
        <property name="adquisitiondate" value="1/1/2002"/>
        <property name="adquisitionprice" value="1000.0"/>
      </asset>
      <asset class="bond" id="op2">>
        <belongs-to concept="office" segment="0004"/>
        <belongs-to concept="product" segment="bond"/>
        <property name="issuedate" value="1/1/2000"/>
        <property name="term" value="120"/>
        <property name="nominal" value="1500.0"/>
        <property name="rate" value="0.07"/>
        <property name="ncoupons" value="20"/>
        <property name="adquisitiondate" value="1/1/2003"/>
        <property name="adquisitionprice" value="1500.0"/>
      </asset>
    </client>
    <client rating="B" sector="S2" name="ATT" id="000002">
      <belongs-to concept="sector" segment="S2"/>
      <belongs-to concept="size" segment="big"/>
      <asset class="bond" id="op3">
        <belongs-to concept="office" segment="0003"/>
        <belongs-to concept="product" segment="bond"/>
        <property name="issuedate" value="1/1/2000"/>
        <property name="term" value="120"/>
        <property name="nominal" value="1500.0"/>
        <property name="rate" value="0.07"/>
        <property name="ncoupons" value="5"/>
        <property name="adquisitiondate" value="1/1/2003"/>
        <property name="adquisitionprice" value="1500.0"/>
      </asset>
    </client>
  </portfolio>

Observese como no se define la pertenencia del cliente al segmento client ya que se realiza de forma implícita (lo mismo con la pertenencia de los activos). En el caso de intentar asignar un cliente a una segmentación de tipo activo (o viceversa) el fichero de configuración será rechazado.

Client Attributes

Name Description Type
rating rating inicial del cliente a fecha de inicio de la simulación. Debe ser un identificador de rating definido en el apartado de ratings. string
sector sector al que pertenece el cliente. Debe ser un identificador de sector definido en el apartado de sectores. string
name nombre del cliente. Usaado solamente a título descriptivo. string
id identificador del cliente. Debe ser un identificador único. string

Belongs-To Attributes

Name Description Type
concept identificador de segmentación al que pertenece el cliente o activo. Si se pretende indicar la pertenencia a un segmento de un cliente el segmento deberá ser de tipo 'client', si se está indicando la pertenencia de un segmento de un activo, el segmento deberá ser de tipo 'asset'. string
segment identificador del segmento. El identificador debe constar como segmento en la segmentación indicada en el atributo concept. Véase apartado segmentations. string

Asset Attributes

Name Description Type
class classe usada para simular el comportamiento del activo. Actualmente las classes existentes son:
bond. Bono. 		string
id identificador de activo. Debe ser un identificador único. string

Property Attributes

Name Description Type
name nombre de la propiedad. Véase la documentación del activo para determinar que propiedades deben espeficarse. string
value valor de la propiedad. Véase la documentación del activo para determinar que propiedades deben espeficarse. string
